A. Propuesta funcional

Cómo funciona la aplicación de principio a fin

1 · Captura de documentos
  • Subida manual de PDF o imagen desde escritorio o móvil (drag & drop y selector de archivo).
  • Escaneo con cámara del móvil directamente desde la app web (PWA con acceso a MediaDevices API).
  • Importación de facturas adjuntas desde Gmail u Outlook mediante OAuth2; el sistema detecta automáticamente adjuntos PDF en correos recibidos.
  • Subida múltiple simultánea: hasta 20 facturas en un solo lote.
2 · Procesamiento OCR e IA
  • PDF nativos: extracción directa de texto mediante pdf-parse; no requiere OCR.
  • PDF escaneados e imágenes: preprocesado con Sharp (contraste, escala de grises, umbralización) y OCR con Tesseract.js o Google Cloud Vision.
  • Extracción de campos mediante GPT-4o con prompt estructurado que devuelve JSON validado con Zod.
  • Campos extraídos: CIF, nombre, fecha, número de documento, base imponible, tipo IVA, cuota IVA, tipo retención, importe retención, total, forma de pago.
3 · Validación y revisión manual
  • Pantalla de revisión que muestra la imagen original junto a los datos extraídos, campo a campo editable.
  • Indicadores visuales de confianza por campo (alto / medio / bajo).
  • Detección de duplicados por combinación CIF + número de documento + importe.
  • Alertas por campos obligatorios vacíos o valores inválidos (CIF con formato incorrecto, fecha futura, etc.).
4 · Clasificación y guardado
  • Cada factura se asigna al cliente correcto; si no se detecta automáticamente, se solicita selección manual.
  • El documento original se sube a la carpeta correspondiente en Google Drive: Asesoría › Cliente › AAAA-MM.
  • El registro se añade como nueva fila al Excel del cliente en Drive (o Google Sheets, según configuración).
  • Estado de la factura: pendiente → revisado → procesado. Cualquier error queda registrado con descripción.

B. Arquitectura técnica

Capas del sistema y responsabilidades

Frontend
  • Next.js 14 (App Router) — PWA con soporte móvil, drag & drop, acceso a cámara.
  • Tailwind CSS para estilos; React Hook Form + Zod para validación de formularios.
  • Visor de PDF inline con react-pdf; editor de campos con resaltado de confianza.
Backend (API)
  • Node.js + Fastify — rutas REST para subida, procesamiento, consulta y exportación.
  • Cola de procesamiento asíncrona con BullMQ + Redis para OCR e IA.
  • Google Drive API v3 + Google Sheets API v4 para escritura en Drive y Excel.
OCR e IA
  • Google Cloud Vision API para OCR de alta precisión en imágenes y PDF escaneados.
  • GPT-4o (OpenAI API) para extracción estructurada de campos contables desde texto OCR.
  • Fallback: Tesseract.js para procesamiento local sin coste.
Base de datos
  • PostgreSQL (Supabase o Railway) — almacenamiento relacional de clientes, facturas, estados e incidencias.
  • Prisma ORM para migraciones y consultas tipadas.
  • Redis para caché de sesiones y cola de jobs.
Integraciones correo
  • Gmail API (OAuth2) — lectura de adjuntos PDF en bandeja de entrada del cliente.
  • Microsoft Graph API — soporte para cuentas Outlook/365.
  • Polling cada 15 minutos o webhook push si está disponible.
Infraestructura
  • Frontend en Vercel; backend en Railway o Render.
  • Autenticación con NextAuth.js (JWT + sesiones).
  • Almacenamiento temporal: S3-compatible (Supabase Storage o Cloudflare R2).

C. Flujo de automatización

Paso a paso desde que llega la factura hasta que queda registrada

  • 1
    Entrada — La factura llega por subida manual (PDF/imagen), escaneo con cámara del móvil o importación desde Gmail/Outlook.
  • 2
    Almacenamiento temporal — El archivo se guarda en Supabase Storage con un ID único y se encola un job en BullMQ.
  • 3
    Pre-procesado — Si es imagen o PDF escaneado, Sharp aplica contraste y binarización para mejorar la calidad del OCR.
  • 4
    OCR — Google Cloud Vision extrae el texto del documento. PDFs nativos se procesan directamente con pdf-parse.
  • 5
    Extracción IA — GPT-4o recibe el texto OCR y un prompt estructurado; devuelve JSON con todos los campos de la factura.
  • 6
    Validación automática — Zod valida el JSON; se verifica formato de CIF, coherencia base + IVA + total, y se buscan duplicados en la BD.
  • 7
    Revisión manual — El contable ve la imagen original y los campos extraídos lado a lado. Puede editar, confirmar o rechazar.
  • 8
    Asignación de cliente — El sistema asocia la factura al cliente correcto (por CIF o selección manual). Estado cambia a "revisado".
  • 9
    Subida a Drive — El archivo original se sube a Asesoría › ClienteX › AAAA-MM usando Google Drive API. Se crea la carpeta si no existe.
  • 10
    Registro en Excel — Se añade una fila al archivo registro_facturas_clienteX.xlsx con los 11 campos definidos. Estado: "procesado".
  • 11
    Trazabilidad — Se registra en la BD: origen, fecha de procesamiento, usuario y estado final. Si hay error, se crea incidencia.

E. Estructura en Google Drive

Árbol de carpetas generado automáticamente por cliente y mes

📁 Asesoría LilitaC ├── 📁 Cliente A — García López, S.L. │ ├── 📁 2026-01 │ │ ├── 📄 FAC-2026-0042_ProveedorX.pdf │ │ └── 📄 FAC-2026-0043_ProveedorY.pdf │ ├── 📁 2026-02 │ ├── 📁 2026-03 │ └── 📊 registro_facturas_GarciaLopez.xlsx │ ├── 📁 Cliente B — Martínez Ruiz (Autónomo) │ ├── 📁 2026-01 │ ├── 📁 2026-02 │ └── 📊 registro_facturas_MartinezRuiz.xlsx │ └── 📁 Cliente C — Pérez Torres (Módulos) ├── 📁 2026-01 └── 📊 registro_facturas_PerezTorres.xlsx
Lógica de creación automática
  • Al procesar la primera factura de un cliente nuevo se crea su carpeta raíz y el archivo Excel.
  • Al procesar la primera factura de un mes nuevo para ese cliente se crea la subcarpeta AAAA-MM.
  • Los IDs de carpeta Drive se guardan en la BD para evitar duplicar carpetas en llamadas posteriores.
  • El nombre de las facturas sigue el patrón: NUM-DOC_NombreEmisor_AAAAMMDD.pdf

Hojas del archivo
  • Registro completo — histórico de todas las facturas, con filtros por mes, estado y forma de pago.
  • Resumen mensual — tabla dinámica auto-generada: total bases, IVA, retenciones y neto por mes.
  • Incidencias — facturas con errores, duplicados detectados o campos faltantes.
Validaciones y formato
  • Columnas numéricas (base, IVA, retención, total) con formato moneda € y 2 decimales.
  • Columna Fecha con formato DD/MM/AAAA.
  • Columna Forma de pago con validación de lista: solo contado o banco.
  • Columna Estado con formato condicional: verde = procesado, amarillo = pendiente, rojo = error.
  • Fila de totales al pie con suma de cada columna numérica.

G. Reglas de negocio

Validaciones, restricciones y normas del sistema

  • Sin duplicados: hash calculado sobre CIF + número de documento + importe total. Si ya existe, se alerta y no se guarda.
  • CIF válido: validación de formato español (letra + 7 dígitos + control) y NIF extranjero (prefijo país + dígitos).
  • Campos obligatorios: CIF, nombre, fecha, número de documento, base imponible y total. Si faltan, el estado queda en "pendiente revisión".
  • Coherencia matemática: se verifica que base + IVA − retención ≈ total (tolerancia ±0.02 € por redondeos).
  • Fechas normalizadas: cualquier formato de fecha (DD/MM/AA, MM-DD-YYYY, etc.) se convierte a ISO 8601 (AAAA-MM-DD).
  • Importes normalizados: comas decimales convertidas a punto, miles separados eliminados, valores siempre positivos.
  • Forma de pago: solo dos valores permitidos — contado o banco. Cualquier otro valor detectado por OCR se muestra para revisión manual.
  • IVA y retención a 0: se registran aunque sean 0%, ya que tienen valor fiscal (exención, no sujeta, etc.).
  • Facturas rectificativas: si el número de documento contiene "R" o "RECT", se marcan con flag especial y se vinculan a la factura original.
  • No sobreescribir: las facturas ya en estado "procesado" no pueden modificarse; solo se permite añadir una incidencia o crear una versión corregida.
